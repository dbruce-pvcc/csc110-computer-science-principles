Chapter 5: Programming Algorithms & Software Testing

Programming Algorithms

    Phases of computer program development:

  • Phase 1: Understand the problem
  • Phase 2: Design an algorithm to solve the problem
  • Phase 3: Write the program code in appropriate language based on the algorithm
  • Phase 4: Enter the program and save
  • Phase 5: Execute the program
  • Phase 6: Test the program, find errors and debug, then retest
  • Phase 7: Document and evaluate the program
    •

    Programming structures:

  • Sequential structure: Top-down execution
  • Decision structures: if/then/else
  • Iterative (looping):
    • while loop: continue looping as long as something is true
    • repeat/until loop: continue looping until something is true
    • for loop: loop a given number of times
  • Recursive Structures:
    • Repeat a set of instructions as a subtask of itself
    • Repetition continues until:
      • degenerative case, which is a termination condition, is met (degenerative case is also called a "base case")

    Two classic programming/database problems:

  • Sorting: Descending order/Ascending order
    • Sorting numbers
    • Sorting strings (in accordance with ASCII values)
  • Searching: find specific data that meet a certain criteria
    • In database:
      • Filters: temporary search
      • Queries: search criteria are saved so they don't have to be recreated
  • Two popular search techniques:
    • Sequential search: search from top to bottom or bottom to top, one item at a time
    • Binary search:
      • only works if data is sorted first
      • start in the middle, then go to half of that, etc.

    Two classic program goals:

  • Efficiency: Program takes minimal time and/or resources
  • Correctness: Program runs in expected ways and works for all expected conditions

Software testing and verification

    Kinds of test:

    Acceptance testing: Verify if the whole system works as intended.
    Integration testing: Ensure the software components or functions operate together.
    Unit testing: Ensure each software unit performs as expected. A unit is the smallest testable component of an application.
    Functional testing: Check functions by emulating business scenarios and base on functional requirements. Black-box testing is a common way to verify functions.
    Performance testing: Test how the software performs under different workloads. For example, load testing evaluates performance under real-life load conditions.
    Regression testing: Check if new features break or degrade functionality. Sanity testing is used to verify menus, functions and commands at the surface level, when a full regression test cannot be done.
    Stress testing: Test how much strain a system can take before it fails. This is a type of non-functional testing.
    Usability testing: Validate how well a customer can use a system or web application to complete a task.

    Software testing is done to detect:

    Architectural flaws: Results from inappropriate design choices in the early stages of software development, incorrect implementation of security patterns, or degradation of security architecture over time.
    Poor design decisions: Poor understanding of the software development practices and lifecycle by the management or the designers themselves.
    Invalid or incorrect functionality: Software does not behave as intended by the developer.
    Security vulnerabilities: Defects in the software that can allow an attacker to gain control of a system.
    Scalability issues: Inability to to grow or shrink the software. In many cases it is the software's ability to handle increased workloads while adding or removing users with minimal cost impact.

    Testing best practices:

    Continuous testing: Test each build as it becomes available. Relies on test automation that is integrated with the deployment process. Enables software validation in realistic test environments earlier in the process. and improves design and reduces risks.
    Configuration management: When organizations centrally maintain test assets and track what software builds to test. Teams gain access to assets such as code, requirements, design documents, models, test scripts and test results.
    Service virtualization: Service virtualization simulates the services and systems that are missing or not yet completed. This enables teams to reduce dependencies and test sooner.
    Defect or "bug tracking": Monitor defects to measure and improve quality. Automated tools allow teams to track defects, measure their scope and impact, and uncover related issues.
    Metrics and reporting: Regular monitoring and reporting relationships between test, development, and other project elements enables team members to share status, goals, and test results to see the overall health of a project.
